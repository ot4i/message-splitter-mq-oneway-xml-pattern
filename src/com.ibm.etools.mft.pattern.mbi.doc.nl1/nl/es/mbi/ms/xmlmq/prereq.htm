Tareas que deben completarse antes de aplicar el patrón Divisor de mensajes para WebSphere MQ: unidireccional (para XML)

Antes de aplicar el patrón para producir una instancia de patrón, debe determinar cuáles de las tareas siguientes son necesarias. Si es preciso, realice las tareas siguientes:

  • Seleccione un nombre exclusivo para la instancia de patrón. Puede utilizar el patrón muchas veces y crear diversas instancias diferentes, que se pueden distinguir por sus nombres exclusivos. Identifique el fin de la aplicación de patrón para que pueda decidir un nombre exclusivo adecuado o significativo.
  • Seleccione las colas y los gestores de colas de WebSphere MQ que desee que utilice la instancia de patrón, por ejemplo, para recibir el mensaje XML. Si está utilizando colas existentes, deben estar registradas. Si desea crear nuevos gestores de colas y colas, debe tomar una decisión sobre los nombres de estos recursos.
  • Decida si es necesaria la validación y defina un conjunto de mensajes o determine el nombre del conjunto de mensajes. Después de generar el patrón, lleve a cabo la tarea de creación del conjunto de mensajes.
  • Identifique cualquier cola y gestor de colas adicionales que este patrón necesite, y decida qué nombres desea utilizar. Tenga en cuenta las políticas, estándares o convenios de denominación que deben aplicarse.
  • Decida si necesita un registro de mensajes entrantes. Si el registro es necesario, debe decidir a qué gestor de colas y cola se enviarán los mensajes de registro. Puede enviar mensajes de registro a una cola de registro general que utilicen varios flujos de mensajes, o puede utilizar una cola específicamente para la instancia de patrón.
  • Decida si hay varios destinos posibles y, si es así, seleccione el enfoque de direccionamiento que va a utilizar.
    • Establezca el parámetro Direccionamiento en Especificar rutas si tiene un número limitado de rutas en que los destinos no cambian con frecuencia.
    • Establezca el parámetro Direccionamiento en Buscar rutas si el conjunto de rutas de destino no está totalmente determinado o puede cambiar a menudo. Si este valor es seleccionado, el nombre del origen de datos debe ser conocido.
  • Decida el enfoque del manejo de errores. Decida si desea elementos de mensaje que no se puedan direccionar correctamente para una retrotracción completa o si basta con escribir el elemento erróneo en una cola para realizar un proceso de excepciones.
  • Decida si desea producir mensajes de error para la instancia de patrón. Todos los datos se conservan sin mensajes de error porque las acciones de error conservan los elementos en una cola para este fin, o retrotraen el flujo de mensajes. El mensaje de entrada se devuelve a la cola de entrada o se envía a una cola de restitución. No obstante, si estas colas no están supervisadas, los mensajes de error pueden ser necesarios como mecanismo de notificación.

    Si elige enviar mensajes de error en la instancia de patrón, puede activar o desactivar esta función configurando el archivo archivador de intermediario (BAR) antes del despliegue.
