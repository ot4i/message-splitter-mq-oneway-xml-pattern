Flux Demande pour le modèle Corrélateur de message pour WebSphere MQ : question-réponse sans persistance

Le flux Demande est exécuté sous la forme d'un flux transactionnel.

Les messages reçus par le flux Demande sont traités de la manière suivante :

  1. Lorsqu'un flux de messages reçoit un message, l'en-tête MQMD est vérifié pour déterminer s'il contient une file d'attente de réponse. Si cette zone est vide, une erreur est générée.
  2. Lorsque le flux de messages reçoit un message, une copie de l'en-tête MQMD est utilisée et modifiée en définissant l'identificateur de corrélation dans l'identificateur du message. Le contenu de l'en-tête de message est enregistré dans une file d'attente pour conserver l'adresse de réponse d'origine. La définition de l'identificateur de corrélation dans l'en-tête enregistré permet de s'assurer que la valeur est correcte lors de l'extraction et de l'ajout de l'en-tête dans la réponse au demandeur et d'extraire le demandeur en fonction de l'identificateur de corrélation.
  3. Le flux de messages inclut le flux secondaire Processeur de demandes. Ce flux secondaire contient uniquement un noeud Passthrough ; par conséquent aucun traitement n'est exécuté. Vous pouvez personnaliser ce flux secondaire dans l'instance du modèle pour ajouter votre propre traitement sans changer la structure du modèle.
  4. L'en-tête MQMD du message entrant est modifié pour remplacer le gestionnaire de file d'attente de réponse et la file d'attente de réponse du demandeur par l'adresse du noeud d'entrée du flux de messages Réponse.
  5. Le message est envoyé à l'application fournisseur.
  6. Le flux de messages peut contenir un flux secondaire, appelé Journal, pour la consignation. Vous pouvez configurer le modèle pour inclure, ou exclure, le flux secondaire Journal.
  7. Si la consignation n'est pas nécessaire, le flux de messages se termine correctement après l'écriture du message dans la file d'attente du fournisseur et la validation de la transaction.
  8. Le flux de messages peut également contenir un flux secondaire, appelé Erreur, pour traiter les erreurs.
  9. Si un dépassement de délai d'attente se produit dans l'application demandeur, les flux de messages ou l'application fournisseur, les messages non persistants expirent.
