Contraintes d'utilisation du modèle Corrélateur de message pour WebSphere MQ : question-réponse sans persistance

Il existe certaines contraintes lorsque vous utilisez le modèle.

Vous ne pouvez utiliser le modèle Corrélateur de message pour WebSphere MQ : question-réponse sans persistance que si toutes les contraintes suivantes sont respectées :

  • Les applications demandeur et fournisseur doivent utiliser WebSphere MQ comme mécanisme de transport.
  • Les applications fournisseurs doivent suivre la convention de copie de l'identificateur du message dans l'identificateur de corrélation pour que le courtier puisse associer les réponses aux demandes d'origine.
  • Les applications demandeur et fournisseur doivent envoyer et accepter des messages compatibles sauf si vous incluez une fonction de transformation dans le flux secondaire Processeur de demandes personnalisable.
  • La fonction métier pour laquelle vous utilisez ce modèle doit pouvoir tolérer les dépassements de délais d'attente et les incidents consécutifs pour recevoir une réponse.
  • Si vous activez la consignation dans le modèle, les messages de journal sont envoyés à une file d'attente. Le traitement de ces messages de journal n'entre pas dans le cadre de ce modèle ; vous devez traiter les messages de consignation aussi efficacement que possible pour l'entreprise. Si vous activez la consignation, les messages de journal sont écrits dans la file d'attente de consignation sous forme de messages persistants, ce qui affecte les performances. Des messages persistants sont utilisés, car la consignation est supposée être utilisée à des fins d'audit. Si la consignation est utilisée pour effectuer des tests ou obtenir uniquement des informations, vous pouvez la désactiver dans les systèmes de production en définissant les propriétés définies par l'utilisateur (UDP), RequestLoggingOn et ResponseLoggingOn.
  • Si vous activez la génération de messages d'erreur dans le modèle, des messages d'erreur sont envoyés à une file d'attente lorsqu'une exception se produit. Le contrôle des erreurs et l'exécution des actions correctives n'entrent pas dans le cadre de ce modèle ; vous devez traiter les messages d'erreur aussi efficacement que possible pour l'entreprise.
  • Le modèle définit son propre format de messages de consignation et d'erreur et ces messages sont supposés être suffisamment efficaces pour répondre aux besoins de votre environnement.
